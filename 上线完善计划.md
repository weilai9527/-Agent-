# 多 Agent 模拟面试系统上线完善计划

## 1. 计划目标

本计划用于将项目从当前的演示与内部测试状态，推进到可公开运营的生产状态。

当前评估：

- 演示完整度：约 82%
- 内部测试完整度：约 70%
- 公开上线准备度：约 50%～55%
- 高并发生产准备度：约 40%～45%

目标状态：

- 公开上线准备度达到 85%～90%
- 管理端、用户端和模型接口具备清晰的权限边界
- 单题评价和综合报告由真实 AI 生成，并具备校准、降级和复核能力
- 核心业务流程具备自动化测试保护
- WebRTC 具备基础监控、重连和容量验证
- 生产环境具备限流、成本控制、告警、备份和回滚能力

## 2. 总体原则

1. 先修复安全风险，再扩展业务功能。
2. 先替换影响产品价值的 Mock 评价和报告，再进行体验优化。
3. 自动化测试随每个阶段同步补充，不在最后集中补测试。
4. 先建立监控和成本保护，再进行真实模型高并发压测。
5. 代码拆分采用渐进方式，在测试保护下逐步迁移，避免一次性重写。
6. 每个阶段必须达到验收标准后，才能进入对应发布节点。

| 阶段 | 工作内容 | 优先级 | 预计人日 | 是否阻塞公开上线 |
|---|---|---:|---:|---:|

| 7 | 部署体系和代码拆分 | P1/P2 | 8～15 | 是 |

- 1 名全栈开发：约 9～13 周
- 2 名开发并行：约 5～7 周
- 3 人小组：约 4～5 周

## 4. 阶段 0：建立项目基线

预计工作量：2～3 人日。

### 4.1 工作内容

- 创建独立的测试和预发布环境配置。
- 区分 development、test、staging、production 环境。
- 锁定 Python 依赖版本。
- 启动时检查必要环境变量，并输出明确错误。
- 对当前 MySQL 数据库执行完整备份。
- 梳理核心接口和业务流程清单。
- 统一 API 错误响应格式。
- 为每个请求生成 Request ID。
- 引入正式数据库迁移机制，建议使用 Alembic。
- 给当前可运行版本创建 Git 标签。
- 标记并隔离不再参与正式构建的原型文件。

### 4.2 交付物

- 多环境配置规范
- 数据库备份文件及恢复说明
- 第一版数据库迁移脚本
- API 和业务流程清单
- 统一错误格式和 Request ID
- 上线检查清单

### 4.3 验收标准

- 开发、测试、预发布环境数据完全隔离。
- 缺少关键配置时服务拒绝启动，并明确指出缺少的配置。
- 空数据库可以通过迁移完整创建。
- 数据库备份可以实际恢复。
- 当前版本能够通过 Git 标签快速恢复。

## 5. 阶段 1：管理端鉴权与权限

预计工作量：5～8 人日。

这是当前最高优先级任务。管理端候选人、面试、报告和系统配置接口必须在公开部署前完成权限保护。

### 5.1 管理员数据模型

建议新增：

- `admin_users`
- `admin_sessions`
- `admin_roles`
- `admin_audit_logs`

管理员角色建议：

| 角色 | 权限范围 |
|---|---|
| 超级管理员 | 管理管理员账号、模型密钥和系统配置 |
| 运营管理员 | 查看用户、面试和报告，处理用户状态 |
| 报告审核员 | 查看和复核报告，不允许修改系统配置 |

管理端应使用独立 Cookie，例如 `admin_session`，不直接复用普通用户会话。

### 5.2 管理端后端鉴权

- 为全部 `/api/admin/*` 业务接口增加管理员登录校验。
- 为敏感接口增加角色校验。
- 未登录统一返回 401。
- 已登录但权限不足统一返回 403。
- 候选人简历、面试内容和报告详情按角色限制。
- 系统模型配置仅允许超级管理员修改。
- 管理员密码失败次数使用 Redis 或数据库持久化限流。

### 5.3 管理端前端

- 增加管理员登录页面。
- 增加退出登录。
- 增加会话过期提示。
- 增加无权限页面。
- 统一处理 401 和 403。
- 支持首次登录修改密码。
- 根据角色隐藏无权访问的菜单和操作。

### 5.4 敏感配置保护

- API Key 只允许更新，不返回完整值。
- 页面仅展示“已配置”或脱敏后的尾部字符。
- 修改模型密钥必须记录审计日志。
- 生产环境优先使用 Secret 管理，不直接编辑 `.env`。
- 禁止运营管理员和审核员修改密钥。

### 5.5 验收标准

- 未登录访问管理数据全部返回 401。
- 权限不足访问敏感功能返回 403。
- 运营管理员不能修改 API Key。
- 所有敏感查看和修改操作都有审计记录。
- 管理端 Cookie 启用 `HttpOnly`、`Secure` 和合理的 `SameSite`。

## 6. 阶段 2：真实 AI 评分和报告

预计工作量：8～12 人日。

### 6.1 统一评价标准

单题评价建议包含以下维度：

- 技术准确性
- 技术深度
- 结构与表达
- 业务理解
- 方案取舍
- 风险意识
- 结果量化
- 岗位匹配度

每个维度需要定义明确的评分锚点，例如：

| 分数 | 定义 |
|---:|---|
| 1～3 | 基本没有回答问题，或存在明显错误 |
| 4～5 | 提到相关概念，但缺少实例和关键细节 |
| 6～7 | 回答完整，具有合理细节和基本取舍 |
| 8～9 | 有深入分析、实际数据、风险和方案权衡 |
| 10 | 达到优秀候选人的完整回答水平 |

### 6.2 真实单题评价服务

建议拆分目录：

```text
backend/src/services/evaluation_service.py
backend/src/providers/
backend/src/schemas/evaluation.py
```

评价输入至少包含：

- 目标岗位
- 面试难度
- 当前 Agent
- 当前问题
- 候选人回答
- 最近对话
- 简历关键内容
- 统一评分标准

模型必须返回结构化 JSON，并使用 Pydantic 校验。

### 6.3 模型供应商降级

建议使用以下链路：

```text
主模型
→ 备用模型
→ 本地规则兜底
```

本地规则兜底结果必须明确标记：

```json
{
  "provider": "local",
  "fallback": true
}
```

不得将规则评分伪装为正式 AI 专业评价。

### 6.4 综合报告服务

报告应综合：

- 全部问题和回答
- 各 Agent 的单题评价
- 候选人的简历和目标岗位
- 回答质量趋势
- 明显优势和短板
- 未覆盖的岗位能力
- 回答之间的一致性
- 面试难度和面试官风格

建议为评价和报告增加以下字段：

```text
provider
model
prompt_version
generation_status
generation_error
generated_at
review_status
reviewed_by
reviewed_at
```

报告生成需要支持：

- pending、running、succeeded、failed 状态
- 失败重试
- 重新生成
- 幂等处理
- Prompt 版本追踪
- 模型版本追踪
- 人工复核

### 6.5 质量校准

- 准备 30～50 组标准面试问答。
- 由人工依据统一标准进行评分。
- 对比 AI 评分和人工评分。
- 调整 Prompt 和各维度权重。
- 保存质量基准数据集，后续模型升级时回归测试。

### 6.6 验收标准

- AI 结构化结果解析成功率不低于 99%。
- 相同回答重复评分的分差处于合理范围。
- AI 与人工评分相差 10 分以内的比例不低于 85%。
- 报告生成成功率不低于 99%。
- 报告失败可以重试，不重复创建记录。
- 页面能够显示供应商、降级状态和复核状态。

## 7. 暂缓项：密码重置邮件能力

当前没有可用的 SMTP 或邮件推送服务，本阶段不启用密码重置入口，也不安排邮件送达、Token 过期和申请次数限制的验收工作。已有代码和数据表保留，但不作为安全内测、公开 Beta 或正式上线的阻塞条件；后续具备邮件服务后再单独恢复排期。

## 8. 阶段 4：隐私、删除与审计

预计工作量：5～8 人日。

### 8.1 用户隐私授权

首次使用前展示并记录：

- 简历处理说明
- AI 供应商处理说明
- 语音采集和转写说明
- 数据保存时间
- 用户的数据删除和导出权利

建议记录：

```text
privacy_version
accepted_at
ip_address
user_agent
```

### 8.2 数据删除

增加接口：

```text
DELETE /api/interviews/:id
DELETE /api/profile/resume
DELETE /api/account
```

账号注销流程：

1. 立即禁用账号。
2. 注销全部 Session。
3. 进入 7～30 天待删除期。
4. 后台任务执行物理删除或匿名化。

### 8.3 数据导出

建议增加：

```text
POST /api/account/export
GET /api/account/export/:taskId
```

导出内容包括：

- 个人资料
- 简历内容
- 面试记录
- 语音转写
- 单题评价
- 综合报告

### 8.4 管理审计

审计日志至少包含：

- 管理员 ID
- 操作类型
- 目标对象
- 修改前后摘要
- IP 地址
- User-Agent
- 操作时间
- 成功或失败状态

### 8.5 日志脱敏

日志禁止记录：

- API Key
- Cookie
- Session Token
- 密码重置 Token
- 用户密码
- 完整简历
- 完整语音转写

### 8.6 验收标准

- 用户可以删除单场面试。
- 用户可以申请注销账号。
- 删除完成后管理端不能继续读取个人数据。
- 所有管理端敏感操作均有审计记录。
- 日志扫描不包含 Token、Key 和密码。

## 9. 阶段 5：主流程自动化测试和 CI

预计工作量：6～10 人日。

测试应与阶段 1～4 同步编写，本阶段用于补齐测试体系和持续集成。

### 9.1 后端测试

使用 pytest 和独立测试数据库，覆盖：

- 注册、登录和退出
- 登录限流
- Session 过期
- 用户间数据隔离
- 管理员角色权限
- 密码重置 Token
- 简历上传和大小限制
- 简历分析及模型降级
- 创建、开始和结束面试
- Agent 顺序与切换
- 消息权限和消息幂等
- 单题评价重复创建
- 报告失败和重试
- 用户数据删除和账号注销
- SDP 参数校验和供应商异常

### 9.2 前端测试

建议使用 Vitest 和 React Testing Library，覆盖：

- 登录和注册表单
- 简历上传和分析状态
- 面试配置校验
- WebRTC 状态切换
- 网络错误提示
- 报告加载和失败重试
- 管理端菜单和角色权限

### 9.3 E2E 测试

建议使用 Playwright 覆盖主流程：

```text
注册/登录
→ 填写资料
→ 上传简历
→ 创建面试
→ 提交回答
→ 生成评价
→ 结束面试
→ 生成报告
→ 查看历史和能力统计
```

WebRTC 自动化测试使用模拟媒体文件，不在每次 CI 中调用真实云端模型。

### 9.4 CI 流程

每次提交执行：

```text
Python 编译
→ 后端单元测试
→ 前端单元测试
→ 候选人端构建
→ 管理端构建
→ API 集成测试
→ 数据库迁移检查
→ 安全和依赖扫描
```

### 9.5 验收标准

- 关键后端模块测试覆盖率不低于 80%。
- 核心前端流程覆盖率不低于 60%。
- 至少一条完整 E2E 流程稳定通过。
- PR 测试失败时禁止合并。
- 测试环境不能连接生产数据库和真实生产密钥。

## 10. 阶段 6：限流、成本、监控和 WebRTC 稳定性

预计工作量：8～12 人日。

### 10.1 接口限流

生产环境建议使用 Redis 实现分布式限流，维度包括：

- IP
- 用户 ID
- 接口类型
- 模型供应商
- 同时进行的面试数量

重点限制接口：

- 注册和登录
- 简历分析
- AI 追问
- AI 评价
- 报告生成
- WebRTC SDP
- TTS

### 10.2 成本统计

建立 AI 调用记录表，建议字段：

```text
user_id
interview_id
provider
model
request_type
input_tokens
output_tokens
audio_seconds
estimated_cost
latency_ms
status
created_at
```

支持：

- 用户每日额度
- 单场面试额度
- 全局日预算
- 供应商预算告警
- 超额后暂停或降级

### 10.3 可观测性

HTTP 指标：

- QPS
- P50、P95、P99 延迟
- 4xx、5xx 比例
- 数据库连接数
- 慢查询

AI 指标：

- 调用成功率
- 首 Token 耗时
- 总耗时
- Token 使用量
- 模型降级次数
- 估算费用

WebRTC 指标：

- SDP 成功率
- 建连成功率
- 建连耗时
- DataChannel 打开率
- 远端音轨到达率
- 异常断线率
- 通话持续时间
- 重连次数

可选技术：

- Sentry：错误追踪
- Prometheus + Grafana：指标监控
- OpenTelemetry：链路追踪
- 结构化 JSON 日志：问题定位

### 10.4 WebRTC 状态机和重连

前端建议建立明确状态机：

```text
idle
→ requesting_permission
→ signaling
→ connecting
→ connected
→ reconnecting
→ failed
→ closed
```

增加：

- SDP 建连超时
- DataChannel 打开超时
- 指数退避重试
- 重新创建 PeerConnection
- 一键重新连接
- 网络恢复监听
- 页面后台挂起后的恢复
- 防止旧连接事件污染新连接

如果供应商支持则增加 ICE Restart；不支持时重新创建完整会话。

### 10.5 压测方案

信令压测：

- 50 并发
- 100 并发
- 300 并发
- 重点压测鉴权、数据库读取和 SDP 代理

长稳测试：

- 20～50 路真实通话
- 持续 2～4 小时
- 检查内存、连接、断线和消息重复

弱网测试：

- 100～300ms 延迟
- 1%～10% 丢包
- Wi-Fi 和移动网络切换
- 短暂断网
- 浏览器后台挂起

### 10.6 验收标准

| 指标 | 目标 |
|---|---:|
| SDP 请求成功率 | ≥99.5% |
| WebRTC 建连成功率 | ≥99% |
| 建连 P95 | ≤5 秒 |
| 30 分钟异常断线率 | ≤1% |
| DataChannel 重复入库 | 0 |
| API 5xx 比例 | ≤0.1% |
| AI 调用记录完整率 | 100% |
| 超预算保护 | 能自动触发 |

## 12. 发布节点

### 12.1 节点 A：安全内测版

必须完成：

## 13. 正式上线门槛

满足以下条件后才建议正式公开上线：

- [ ] 管理接口未经授权全部返回 401 或 403。
- [ ] 不再使用 Mock 结果作为正式 AI 评价。
- [ ] AI 评价完成质量校准。
- [ ] 用户可以删除面试和申请删除个人数据。
- [ ] 管理端敏感操作全部记录审计日志。
- [ ] 完整主流程 E2E 测试稳定通过。
- [ ] WebRTC 建连成功率不低于 99%。
- [ ] WebRTC 经过长时间和弱网测试。
- [ ] AI 接口具备用户级限流和成本上限。
- [ ] 生产错误可以主动告警。
- [ ] 数据库备份完成恢复演练。
- [ ] 生产部署失败可以快速回滚。
- [ ] 日志不包含密码、Token、API Key 和完整简历。
